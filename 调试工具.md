# CUDA 调试工具使用指南

> 适用环境：Linux

---

## 一、编译准备

调试前需要让 CUDA 代码保留调试信息：

```bash
# 基本调试编译（保留行号信息）
nvcc -g -lineinfo -o myapp myapp.cu

# 完整调试编译（保留源码级调试，性能会下降）
nvcc -g -G -o myapp myapp.cu

# CMake 项目中添加调试选项
cmake .. -DCMAKE_BUILD_TYPE=Debug \
         -DCUDA_ARCHITECTURES="75;80;86" \
         -DCMAKE_CUDA_FLAGS="-g -lineinfo"
```

| 编译选项 | 作用 | 性能影响 |
|---------|------|---------|
| `-g` | 生成调试符号 | 小 |
| `-lineinfo` | 保留 SASS → 源码行号映射 | 小 |
| `-G` | 完整设备代码调试（device-side debug） | 大（可能 2-10x 变慢） |
| `-O0` | 禁用优化 | 大 |

**建议**：先用 `-g -lineinfo` 跑 `compute-sanitizer`；需要 `cuda-gdb` 单步 kernel 时再加 `-G`。

## 二、compute-sanitizer（内存/竞态/同步检查）

### 2.1 基本用法

```bash
compute-sanitizer --tool memcheck ./myapp [args...]
```

### 2.2 四大工具

| 工具 | 检查内容 |
|------|---------|
| `memcheck` | 内存越界、非法访问、未初始化读、错误释放 |
| `racecheck` | 共享内存数据竞争（需按提示重跑） |
| `initcheck` | 局部/寄存器内存未初始化使用 |
| `synccheck` | __syncthreads() 漏用/错用 |

#### memcheck（最常用）

```bash
# 基本内存检查
compute-sanitizer --tool memcheck ./myapp

# 保存报告到文件
compute-sanitizer --tool memcheck --log-file report.txt ./myapp

# 抑制已知误报（适用于第三方库）
compute-sanitizer --tool memcheck --suppress suppression_file.txt ./myapp

# 只看错误汇总（适合 CI）
compute-sanitizer --tool memcheck --print-limit 0 --fatal 0 ./myapp 2>&1 | tail -5

# 显示详细调用栈
compute-sanitizer --tool memcheck --show-backtrace device,host ./myapp
```

**常见错误类型及含义：**

| 错误 | 含义 | 典型原因 |
|------|------|---------|
| `out-of-bounds` on `global` | 全局内存越界 | 数组下标未检查、pitch/stride 算错 |
| `out-of-bounds` on `local` | 局部内存越界 | 栈数组越界、寄存器溢出到 local 后越界 |
| `out-of-bounds` on `shared` | 共享内存越界 | 动态 shared 大小不够、索引超限 |
| `misaligned` | 非对齐访问 | 强制类型转换导致指针对齐不满足 |
| `invalid` on `global` | 非法全局内存地址 | 空指针、已释放内存、野指针 |
| `uninitialized` | 未初始化读取 | 局部变量/寄存器未赋值就使用 |
| `heap-use-after-free` | 释放后使用 | cudaFree 后继续访问 |
| `heap-double-free` | 重复释放 | cudaFree 同一块内存两次 |

#### racecheck

```bash
# 第一次运行：检测是否存在竞争（快速扫描）
compute-sanitizer --tool racecheck ./myapp

# 如果提示有竞争，按提示加上竞态检测参数重新运行
compute-sanitizer --tool racecheck --racecheck-report analysis ./myapp
```

#### initcheck

```bash
compute-sanitizer --tool initcheck ./myapp
```

#### synccheck

```bash
compute-sanitizer --tool synccheck ./myapp
```

### 2.3 实战排查流程

```
第一步：不加任何参数跑 memcheck
  compute-sanitizer --tool memcheck ./myapp
  → 如果没报错，基本排除内存越界

第二步：有报错时，开启行号信息重编
  nvcc -g -lineinfo -o myapp myapp.cu
  compute-sanitizer --tool memcheck ./myapp
  → 报错会带源码行号

第三步：针对报错类型修复
  - out-of-bounds → 检查 idx < n 边界
  - misaligned    → 检查指针类型和数据类型
  - uninitialized → 初始化变量

第四步：如果涉及多线程同步问题
  compute-sanitizer --tool synccheck ./myapp
  compute-sanitizer --tool racecheck ./myapp
```

### 2.4 常用选项

```bash
# 遇到第一个错误立即退出（适合 CI 或快速定位）
compute-sanitizer --tool memcheck --fatal all ./myapp

# 保存报告到带时间戳的文件
compute-sanitizer --tool memcheck --save all ./myapp

# 同时跑多个检查工具
compute-sanitizer --tool memcheck,racecheck,synccheck ./myapp

# 限制输出数量（避免刷屏）
compute-sanitizer --tool memcheck --print-limit 50 ./myapp

# 只检查特定 kernel（通过正则过滤）
compute-sanitizer --tool memcheck --kernel-regex "my_kernel" ./myapp

# 显示详细调用栈
compute-sanitizer --tool memcheck --show-backtrace device,host ./myapp

# 设置内存检查模式
compute-sanitizer --tool memcheck --access-origin on ./myapp

# 指定要追踪的内存分配
compute-sanitizer --tool memcheck --track-memory-allocations on ./myapp
```

## 三、cuda-gdb（源码级调试器）

### 3.1 启动

```bash
# 直接启动
cuda-gdb ./myapp

# 带参数启动
cuda-gdb --args ./myapp arg1 arg2

# 附加到运行中的进程
cuda-gdb -p <pid>
```

### 3.2 常用命令速查

#### 断点管理

```gdb
# 在主机函数设断点
(gdb) break main.c:42
(gdb) break my_function

# 在 kernel 函数设断点（设备端）
(cuda-gdb) break my_kernel
(cuda-gdb) break my_kernel if threadIdx.x == 0

# 在设备代码指定行设断点
(cuda-gdb) break my_kernel.cu:128

# 查看所有断点
(cuda-gdb) info breakpoints

# 删除断点
(cuda-gdb) delete 1
```

#### 运行控制

```gdb
# 启动程序
(cuda-gdb) run [args]

# 继续
(cuda-gdb) continue

# 单步执行
(cuda-gdb) next          # step over
(cuda-gdb) step          # step into

# 设备端单步
(cuda-gdb) device_next   # device step over
(cuda-gdb) device_step   # device step into

# 运行到当前函数结束
(cuda-gdb) finish
```

#### 查看变量和内存

```gdb
# 打印变量
(cuda-gdb) print var_name
(cuda-gdb) print arr[idx]
(cuda-gdb) print *ptr

# 查看设备内存（指定线程）
(cuda-gdb) print threadIdx
(cuda-gdb) print blockIdx
(cuda-gdb) print var_name

# 以十六进制查看内存
(cuda-gdb) x/16xb &arr[0]

# 查看寄存器
(cuda-gdb) info registers
```

#### 线程/块切换（核心能力）

```gdb
# 列出所有线程块的线程
(cuda-gdb) info threads

# 切换到特定线程块和线程
(cuda-gdb) cuda block(0,0,0) thread(128,0,0)

# 查看当前 warp 信息
(cuda-gdb) info warp

# 焦点在所有线程上切换
(cuda-gdb) focus         # 查看当前焦点
(cuda-gdb) set cuda focus_block(1,0,0)
(cuda-gdb) set cuda focus_thread(0,0,0)
```

#### 调用栈

```gdb
# 查看主机调用栈
(gdb) backtrace

# 查看设备调用栈
(cuda-gdb) cuda backtrace

# 切换调用栈帧
(cuda-gdb) frame 2
```

#### kernel 控制

```gdb
# 在 kernel 入口设断点（所有线程）
(cuda-gdb) break my_kernel

# 只在特定线程命中断点
(cuda-gdb) break my_kernel if (threadIdx.x == 0 && blockIdx.x == 0)

# 自动在每个 kernel 启动时暂停（最常用设置之一）
(cuda-gdb) set cuda break_on_launch kernel

# 取消自动暂停
(cuda-gdb) set cuda break_on_launch none
```

#### 核心调试设置

```gdb
# 关闭 kernel 超时（防止调试时被 TDR 杀掉，非常常用）
(cuda-gdb) set cuda kernel_timeout on

# 在 gdb 内强制同步执行（等同于 CUDA_LAUNCH_BLOCKING=1）
(cuda-gdb) set cuda context_launch_blocking on

# 设置 software breakpoint 数量上限（默认 256）
(cuda-gdb) set cuda software_preempt on

# 查看当前 GPU 和 context 信息
(cuda-gdb) info cuda devices
(cuda-gdb) info cuda contexts

# 查看当前 CUDA 状态
(cuda-gdb) info cuda current

# 关闭分页（避免输出时频繁按回车）
(cuda-gdb) set pagination off
```

### 3.3 cuda-gdb 常见调试场景

#### 场景 1：定位哪个线程导致越界

```gdb
(cuda-gdb) break my_kernel.cu:50    # 怀疑出问题的行
(cuda-gdb) run
# 命中后检查当前线程索引
(cuda-gdb) print threadIdx.x, blockIdx.x, gridDim.x
# 循环到下一个命中
(cuda-gdb) continue
```

#### 场景 2：检查 shared memory 内容

```gdb
(cuda-gdb) break my_kernel.cu:80    # __syncthreads() 之后的行
(cuda-gdb) run
(cuda-gdb) print shared_var[index]
```

#### 场景 3：异步错误定位

```bash
# 先在代码里加 CUDA_LAUNCH_BLOCKING=1，让错误同步到当前 kernel
$ export CUDA_LAUNCH_BLOCKING=1
$ cuda-gdb ./myapp
```

### 3.4 cuda-gdb 使用限制

| 限制 | 说明 |
|------|------|
| 单 GPU | 只能调试一个 GPU 上的程序 |
| 线程数限制 | 断点最多同时挂 256 个线程 |
| 不支持 attach 到 CUDA MPS | 需要单进程模式运行 |
| 编译要求 | 需要至少 `-g -lineinfo`，设备端 step 需要 `-G` |

## 四、联合使用建议

### 4.1 推荐流程

```
1. compute-sanitizer --tool memcheck     → 定位内存错误类型和位置
2. compute-sanitizer --tool synccheck    → 排查同步问题
3. compute-sanitizer --tool racecheck    → 排查共享内存竞争
4. cuda-gdb                              → 源码级单步调试，理解具体逻辑
```

### 4.2 性能开销对比

| 工具 | 启动开销 | 运行时开销 | 适合场景 |
|------|---------|-----------|---------|
| `memcheck` | 中 | 高（5-30x） | 全面排查内存问题 |
| `initcheck` | 中 | 中 | 查未初始化变量 |
| `racecheck` | 低 | 低 | 查共享内存竞争 |
| `synccheck` | 低 | 中 | 查 __syncthreads 问题 |
| `cuda-gdb -G` | 低 | 高 | 单步调试、逻辑分析 |

## 五、环境变量

```bash
# 确保 CUDA 工具链在 PATH 中
export PATH=/usr/local/cuda/bin:$PATH
export CUDA_HOME=/usr/local/cuda

# 强制同步执行（排查异步错误时非常有用）
export CUDA_LAUNCH_BLOCKING=1

# 指定使用的 GPU
export CUDA_VISIBLE_DEVICES=0
```

## 六、常见问题速查

| 问题 | 解决方案 |
|------|---------|
| `compute-sanitizer: command not found` | 安装 CUDA Toolkit 或检查 PATH |
| 报错没有源码行号 | 重新编译加 `-g -lineinfo` |
| `cuda-gdb` 无法启动程序 | 确认不是通过 MPS 运行 |
| 报错位置和实际代码不对应 | 检查编译优化级别，用 `-O0 -g` 重编 |
| `memcheck` 运行太慢 | 减少 `--print-limit`，或只测试小数据集 |
| `racecheck` 需要重跑多次 | 这是正常的，按提示操作即可 |
| release 版找不到问题 | 切换到 Debug 编译 `-O0 -g -G` |
